= Architecture Principles

Use this when the user asks why a rewrite is necessary.

== Principles

* AF5 makes message handling more explicit.
* Event-sourced state is modeled as entities, not opaque aggregates.
* Context is passed through APIs instead of hidden thread-local state.
* Processor configuration is component-oriented and named.
* Storage semantics must be explicit during migration; defaults are not a safe
  proxy for AF4 behavior.

== Migration Implications

[cols="1,2"]
|===
|Old habit |AF5 implication

|Thread-local unit of work
|Use method parameters / `ProcessingContext`.

|Implicit aggregate creation
|Make entity creation behavior test-backed.

|Starter/default event store
|Declare the aggregate-based engine when preserving legacy storage.

|Central configurer catch-all
|Use focused messaging/event-sourcing configuration APIs.

|Saga as automatic long-running workflow
|Record a design decision; do not auto-port.
|===

== Rule

When preserving behavior conflicts with a tempting AF5 default, preserve
behavior and record the reason.
